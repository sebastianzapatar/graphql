GraphQL vs REST API

Por que elegir GraphQL para tus APIs modernas

Ingenieria de Sistemas y Computacion • 2026

Que es REST?

REST (Representational State Transfer) es un estilo arquitectonico para APIs. Usa multiples endpoints con metodos HTTP estandar.

🔗

Multiples Endpoints

Cada recurso tiene su URL: /users, /posts, /comments

🔄

Sin Estado

Cada peticion es independiente y autocontenida

⚙

Metodos HTTP

GET, POST, PUT, DELETE para CRUD

Que es GraphQL?

Lenguaje de consultas para APIs desarrollado por Facebook (2012, open-source 2015). Un solo endpoint, el cliente pide exactamente lo que necesita.

🎯

Un Solo Endpoint

Todas las operaciones van a /graphql

📋

Schema Tipado

Contrato fuertemente tipado entre cliente y servidor

⚡

Consultas Flexibles

El cliente define que campos recibir

REST vs GraphQL: Visual

  • REST: Multiples endpoints, cada recurso es una URL
  • GraphQL: Un solo endpoint, el cliente decide los datos
  • REST: Respuestas fijas predefinidas por el servidor
  • GraphQL: Respuestas dinamicas definidas por el cliente

Over-fetching y Under-fetching

Over-fetching

REST retorna TODOS los campos aunque solo necesites 2.

Under-fetching

Para datos relacionados necesitas multiples peticiones REST.

Por que usar GraphQL?

🚀

Eficiencia en Red

Una sola peticion obtiene todos los datos incluyendo relaciones anidadas.

🛠

Developer Experience

Schema tipado, auto-documentacion, GraphiQL para explorar la API.

📱

Ideal para Mobile

Conexiones lentas se benefician al recibir solo lo necesario.

📈

Sin Versionamiento

Nuevos campos no rompen clientes. Los obsoletos se marcan deprecated.

Operaciones GraphQL: Schema

El schema define el contrato de la API. Es el corazon de GraphQL.

# Definicion del tipo - como una tabla en una BD type Producto { id: ID! # ! = campo obligatorio nombre: String! precio: Float! stock: Int! categoria: String # Sin ! = campo opcional } # Queries = operaciones de lectura (GET) type Query { productos: [Producto!]! productoPorId(id: ID!): Producto } # Mutations = operaciones de escritura (POST/PUT/DELETE) type Mutation { crearProducto(nombre: String!, precio: Float!): Producto! eliminarProducto(id: ID!): Boolean! }

Operacion: Query (Lectura)

Las queries son el equivalente a GET en REST. Solo leen datos.

Peticion del cliente

# Pido solo nombre y precio, nada mas query { productos { nombre precio } } # Buscar por ID con mas campos query { productoPorId(id: "abc123") { nombre precio stock categoria } }

Respuesta del servidor

{ "data": { "productos": [ { "nombre": "Laptop HP", "precio": 3500000 }, { "nombre": "Mouse Logitech", "precio": 85000 } ] } }

Operacion: Mutation (Escritura)

Las mutations son el equivalente a POST PUT DELETE en REST.

Crear producto CREATE

mutation { crearProducto( nombre: "Laptop HP Pavilion" descripcion: "16GB RAM, 512 SSD" precio: 3500000 stock: 15 categoria: "Electronica" ) { id # Retorna el ID generado nombre precio } }

Actualizar producto UPDATE

mutation { actualizarProducto( id: "abc123" precio: 2999000 stock: 8 ) { id nombre precio # Precio actualizado stock # Stock actualizado } }

Mutation: Eliminar + Variables

Eliminar DELETE

mutation { eliminarProducto(id: "abc123") } # Respuesta: { "data": { "eliminarProducto": true } }

Usando Variables

# Query con variables (buena practica) query BuscarPorPrecio( $min: Float!, $max: Float! ) { productosPorPrecio( min: $min, max: $max ) { nombre precio categoria } } # Variables (JSON): { "min": 10000, "max": 500000 }

REST vs GraphQL: 3 peticiones vs 1

REST 3 peticiones

// Peticion 1: usuario GET /api/users/1 // Peticion 2: posts del usuario GET /api/users/1/posts // Peticion 3: seguidores GET /api/users/1/followers // Total: 3 round-trips al servidor // Cada respuesta trae datos innecesarios

GraphQL 1 peticion

query { user(id: "1") { name email posts { title createdAt } followers { name } } } // Total: 1 round-trip, datos exactos

GraphQL en el Mundo Real

Empresas que lo usan

Facebook, GitHub, Shopify, Twitter, Airbnb, Netflix en produccion con millones de peticiones.

Caso GitHub

Migro de REST v3 a GraphQL v4 porque los clientes necesitaban datos de multiples recursos en una sola vista.

Cuando NO usarlo

APIs simples CRUD, microservicios internos, o cuando el cache HTTP es critico.

Mejor Practica

Usar como capa de agregacion sobre microservicios REST (patron BFF).

Fuente: r/graphql - Reddit Community

Arquitectura: Java + GraphQL + MongoDB

  • Java 21 con Spring Boot 3.x
  • Spring for GraphQL como motor
  • MongoDB en Docker (localhost:27017)
  • Lombok para reducir boilerplate
  • Schema-first: archivo .graphqls + resolvers
  • GraphiQL: UI web para probar queries

Codigo: Modelo con Lombok

@Document(collection = "productos") // Coleccion MongoDB @Data @Builder // Lombok: getters/setters + builder @NoArgsConstructor @AllArgsConstructor // Constructores public class Producto { @Id private String id; // ID generado por MongoDB private String nombre; // Nombre del producto private String descripcion; // Descripcion detallada private Double precio; // Precio en pesos private Integer stock; // Unidades disponibles private String categoria; // Categoria }

Codigo: Controller GraphQL

@Controller @RequiredArgsConstructor // Lombok inyeccion public class ProductoController { private final ProductoService servicio; @QueryMapping // Query: leer public List<Producto> productos() { return servicio.obtenerTodos(); } @QueryMapping // Query: buscar public Producto productoPorId(@Argument String id) { return servicio.obtenerPorId(id).orElse(null); } @MutationMapping // Mutation: crear public Producto crearProducto( @Argument String nombre, @Argument Double precio, ...) { return servicio.crear(nombre, ...); } }

Cuando usar cada uno?

Usa REST cuando...

  • API simple con operaciones CRUD basicas
  • Cache HTTP es critico
  • Tu equipo ya domina REST
  • Datos con estructura predecible y fija
  • Microservicios internos simples

Usa GraphQL cuando...

  • Multiples clientes con diferentes necesidades
  • Datos complejos con relaciones anidadas
  • Reducir peticiones de red es clave
  • La API evoluciona sin romper clientes
  • Tiempo real con subscriptions
